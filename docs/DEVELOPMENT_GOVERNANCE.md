# Lynx 开发治理：三省六部制

> 目标：后续代码开发借鉴“三省六部制”，把需求拆解、方案审查、执行落地、质量验收制度化，减少交互偏差、返工和隐性风险。

## 1. 总原则

后续每个非简单需求都按以下顺序推进：

1. **中书省：拟诏 / 方案起草**
   - 明确用户真实意图、业务边界、交互语义和影响范围。
   - 先调研现有实现，避免凭空设计。
   - 输出可执行计划，标明涉及页面、接口、数据和验证方式。

2. **门下省：封驳 / 审查复核**
   - 对方案做反向审查：是否违背已有交互规则、是否有移动端/PWA遗漏、是否有性能和数据一致性风险。
   - 对不合理方案主动“封驳”，回到中书省重拟。
   - 特别检查：全局操作 vs 单项操作、桌面端 vs 移动端、实时数据 vs 缓存数据。

3. **尚书省：奉行 / 工程执行**
   - 按最小变更集落地代码。
   - 每个文件改动保持风格一致，不做无关重构。
   - 补齐 i18n、路由、API、迁移、样式和测试验证。

## 2. 六部职责映射

### 2.1 吏部：需求与权限

负责确认“谁在什么场景下做什么”。

检查项：

- 需求是否属于页面级操作、记录级操作、后台任务或系统设置？
- 是否需要登录态、权限、只读/可编辑状态区分？
- 是否会影响现有用户路径？
- 是否需要保留已有行为或兼容旧数据？

典型产物：

- 需求边界说明
- 用户路径说明
- 交互入口位置说明

### 2.2 户部：数据、资产与成本

负责数据结构、缓存、性能成本和资源使用。

检查项：

- 是否需要数据库迁移？迁移是否可重复、安全、可回滚思考过？
- 数据来源是实时、缓存、手工录入还是外部 API？
- 是否会造成慢查询、重复拉取、接口风暴？
- 是否需要后端聚合/降采样，避免前端拉取大数据？
- 货币、时区、单位是否一致？

典型产物：

- API 数据结构
- 数据库迁移
- 缓存/降级策略

### 2.3 礼部：交互、视觉与 i18n

负责用户体验、视觉一致性和国际化。

检查项：

- 桌面端、移动端、PWA 是否都可用？
- 操作入口是否符合既有规则：
  - 页面级操作放顶部或移动端三点菜单
  - 单项操作放卡片、左划、详情页或行内按钮
- 是否遮挡表单、破坏抽屉、破坏安全区？
- 是否使用主题变量，支持深色、浅色、玻璃、紫色主题？
- 中英文 i18n key 是否齐全？

典型产物：

- 页面结构
- 组件样式
- i18n 文案
- 移动端/PWA 适配说明

### 2.4 兵部：执行、任务与容错

负责异步流程、后台任务、Agent、重试与失败降级。

检查项：

- 大模型、行情、资讯、推送等外部依赖是否有重试？
- 单个环节失败是否会拖垮整个任务？
- 是否有 fallback、缓存回退、部分成功策略？
- 日志是否能定位失败源？
- 是否避免重复任务、并发冲突和请求取消问题？

典型产物：

- 重试策略
- fallback 策略
- 任务状态与进度事件
- 日志字段

### 2.5 刑部：安全、边界与异常

负责风险、输入校验、错误处理和破坏性操作。

检查项：

- API 是否校验参数类型、范围和必填字段？
- 删除、覆盖、批量操作是否有确认？
- 外部文本是否做 prompt injection 防护？
- Secret 是否掩码显示，避免回写 `****`？
- 错误提示是否可理解，不泄露敏感信息？

典型产物：

- 参数校验
- 错误处理
- 安全过滤
- 确认弹窗

### 2.6 工部：代码、构建与发布

负责工程实现质量和最终验收。

检查项：

- 是否保持最小改动？
- 是否复用现有组件和工具？
- 是否新增必要组件、服务、路由、迁移和文档？
- 是否运行：
  - 语法检查
  - i18n parity
  - `git diff --check`
  - 前端构建
  - 必要的服务端导入/临时数据库验证
- 是否说明已知非阻塞 warning？

典型产物：

- 代码实现
- 构建结果
- 验证记录
- 最终变更清单

## 3. 每次开发的执行模板

后续非简单需求默认按以下模板执行。

### 3.1 中书省：起草计划

```text
目标：
- 用户要解决什么问题？

影响范围：
- 前端页面：
- 组件：
- 后端路由：
- 服务：
- 数据库：
- i18n：

拟定方案：
1. ...
2. ...
3. ...

验证方式：
- ...
```

### 3.2 门下省：封驳检查

```text
审查问题：
- 是否把单项操作误放到页面级入口？
- 是否遗漏移动端/PWA？
- 是否可能遮挡表单或破坏抽屉布局？
- 是否会触发慢请求或重复请求？
- 是否有外部依赖失败后的 fallback？
- 是否需要数据库迁移？
- 是否需要 i18n？

结论：
- 通过 / 驳回重拟
```

### 3.3 尚书省：执行落地

```text
执行步骤：
1. 修改后端数据/API
2. 修改前端组件/页面
3. 补齐 i18n/路由/迁移
4. 跑诊断和构建
5. 总结变更与验证结果
```

## 4. 默认质量门禁

后续改动完成前，默认至少执行：

```zsh
cd /Users/guxiaolong/workspace/lynx
node --check <modified-server-file.js>
git --no-pager diff --check
```

涉及前端时执行：

```zsh
cd /Users/guxiaolong/workspace/lynx/client
npm run build
```

涉及 i18n 时执行 key parity 检查：

```zsh
cd /Users/guxiaolong/workspace/lynx
node /tmp/lynx-i18n-check.mjs
```

涉及数据库迁移或复杂数据计算时，使用临时数据库验证，避免污染真实数据：

```zsh
DB_PATH="$(mktemp -t lynx-test).db" node <test-script>
```

## 5. 特别适用于 Lynx 的既定规则

1. **移动端操作规则**
   - 页面级操作：顶部三点菜单。
   - 单项操作：卡片按钮、左划、详情页操作。
   - 不把“某个资产/某条记录”的操作放到全局菜单。

2. **表单操作规则**
   - 抽屉内优先使用 `AppDrawer #footer`。
   - 通用表单可用 `MobileActionBar`，但不得遮挡输入项。

3. **行情规则**
   - 普通页面预览优先读缓存。
   - 用户明确刷新时才强制实时请求。
   - 手工行情应进入 `price_cache` 并参与趋势图。

4. **AI Agent 规则**
   - 外部请求要有超时、重试、fallback。
   - 单步失败不应默认导致整个 Agent 失败。
   - 进度、数据质量、fallback 状态要可见。

5. **主题规则**
   - 新增颜色优先使用语义变量。
   - 深色、浅色、玻璃、紫色主题都要可读。

6. **PWA 规则**
   - 注意安全区、底部导航、滚动区域和触控目标大小。
   - 避免过重依赖和大体积图表库。
